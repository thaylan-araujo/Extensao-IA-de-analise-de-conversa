    2. Tabelas (todas em public, todas com created_at timestamptz not null default now()):
       - organizations: id uuid pk default gen_random_uuid(), name text not null (D-04: cadastro mínimo, só nome).
       - profiles: user_id uuid pk references auth.users(id) on delete cascade, organization_id uuid not null references organizations(id) on delete cascade, full_name text not null, role user_role not null default 'advogado', status profile_status not null default 'active', removed_at timestamptz. PK em user_id garante D-08 (1 usuário = exatamente 1 organização).
       - invitations: id uuid pk default gen_random_uuid(), organization_id uuid not null references organizations(id) on delete cascade, email text not null (armazenar em lowercase), role user_role not null default 'advogado' com CHECK (role <> 'super_admin'), token_hash text not null unique, expires_at timestamptz not null, status invitation_status not null default 'pending', invited_by uuid references auth.users(id) on delete set null. Índice único parcial em (organization_id, lower(email)) where status = 'pending' — um convite pendente por e-mail por organização.
       - conversations: id uuid pk default gen_random_uuid(), organization_id uuid not null references organizations(id) on delete cascade, profile_id uuid not null references profiles(user_id), wa_chat_id text, contact_name text, updated_at timestamptz not null default now(). (Tabela central da Fase 2 — Pattern 6: criada agora com cascata LGPD; profile_id NÃO cascateia de profiles porque remoção de membro é soft e o histórico sobrevive, D-11.)
       - messages: id uuid pk default gen_random_uuid(), conversation_id uuid not null references conversations(id) on delete cascade, organization_id uuid not null references organizations(id) on delete cascade, sender text not null, content text not null, sent_at timestamptz.

    5. Habilitar RLS em TODAS as 7 tabelas (enable row level security na mesma migration — anti-pattern da pesquisa: tabela sem RLS). Policies, todas com `to authenticated` e helpers envoltos em (select ...) — Pitfall 3:
       - organizations select: id = (select private.current_org_id()) OR (select private.is_super_admin()).
       - profiles select: user_id = (select auth.uid()) OR (organization_id = (select private.current_org_id()) AND (select private.is_gestor())) OR (select private.is_super_admin()). Advogado enxerga só o próprio perfil; gestor enxerga a equipe (AUTH-04).
       - invitations select: organization_id = (select private.current_org_id()) AND (select private.is_gestor()) OR (select private.is_super_admin()).
       - conversations select: organization_id = (select private.current_org_id()) AND ((select private.is_gestor()) OR profile_id = (select auth.uid())) OR (select private.is_super_admin()). Gestor vê tudo da org, advogado só as próprias (criterion 1); gestor também atende com conversas próprias (D-06).
       - conversations insert: with check organization_id = (select private.current_org_id()) AND profile_id = (select auth.uid()).
       - messages select/insert: espelham conversations (select: org do usuário + gestor-ou-dono via exists em conversations; insert: with check org do usuário + conversa pertence ao usuário).
       - diagnostics select: mesma regra de conversations select; SEM policy de insert (só service_role gera diagnóstico — server-side).
       - audit_log select: (select private.is_super_admin()); SEM policies de escrita para authenticated (escrita só via service_role).
